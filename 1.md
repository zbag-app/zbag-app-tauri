### 2) data-model.md “Mnemonic storage options” conflicts with NFR-009 wallet-password encryption requirement

**Problem**

* `spec.md` NFR-009: seed phrase MUST be encrypted at rest using a user-defined wallet password.
* `plan.md` reinforces wallet-password-derived KEK and wrapped DEK, with optional OS keychain “remember unlock”.
* But `data-model.md` currently frames mnemonic storage as “choose one per deployment” including “OS Keychain (preferred)” as a primary storage option, which reads like “keychain-only” protection. That conflicts with “encrypted with wallet password” as a hard requirement.

**Change**

* File: `specs/001-zkore-desktop-wallet/data-model.md`
* Section: `Key Storage Architecture` under “Mnemonic storage options”

Replace the current “choose one per deployment” list with wording that makes wallet-password encryption mandatory whenever anything is persisted, and keychain strictly a convenience layer.

Suggested replacement block (edit as needed, but keep the constraints explicit):

```md
**Mnemonic persistence (v1):**
- If the mnemonic is persisted anywhere, it MUST be encrypted using the user-defined wallet password (per the v1 key hierarchy in plan.md: wallet password → Argon2id KEK → unwrap per-wallet DEK).
- OS keychain "remember unlock" is OPTIONAL and stores only unlock material (DEK or wrapping secret), never as the sole protection for the mnemonic and never as a substitute for per-action re-auth.

**Mnemonic storage modes:**
1. **Encrypted blob on disk (default):** Store an encrypted mnemonic blob in the wallet directory (or other backend-controlled storage). Decrypt requires wallet password (unless auto-unlock is enabled via keychain-stored DEK).
2. **Memory-only mode (optional):** Do not persist mnemonic at rest; user must re-enter it on each app launch.
```

(If you still want to mention keychain as a storage location, clarify it stores DEK/unlock material, not the mnemonic plaintext, and that wallet-password encryption is still the primary protection.)

## Recommendations

These are not strict misalignments, but they will improve clarity and reduce implementation risk.

### A) Add a lightweight traceability matrix

Right now, traceability is implicit (tasks grouped by user story), but you asked for checklist tracing. The simplest improvement is adding a small section to `tasks.md` or a new `specs/001-zkore-desktop-wallet/traceability.md` with:

* User Story → FR IDs covered
* Non-functional requirements → task groups (logging, a11y, encryption, IPC validation)

This can be a short table and will make audits and reviews easier.

### B) Clarify memo length semantics

`data-model.md` says memo max 512 bytes; `ipc-v1.ts` has MEMO_TOO_LONG but does not define byte vs character length.
Add one sentence in either `data-model.md` or `ipc-v1.ts` clarifying:

* Memo limit enforcement is by byte length after encoding to the on-chain memo format, not by JS string length.

### C) Consider dedicated error codes for “network unsupported” cases

You currently plan to disable swaps on testnet; you can implement this using an existing stable code, but UX is better with a dedicated code. Consider adding:

* `SWAP_UNSUPPORTED_NETWORK` (or similar) rather than overloading `SWAP_FAILED`.

Only do this if you are comfortable changing the contract now; otherwise document which existing code will be used.

### D) Verify duplicated docs under `docs/`

You have `docs/constitution.md`, `docs/plan.md`, `docs/spec.md`, `docs/task.md` but their contents were not included here. Given `AGENTS.md` says to keep `docs/` and `specs/` in sync, you should run a manual diff and ensure at least these topics are consistent:

* “View seed phrase” amendment (FR-008b, constitution v1.1.0)
* Encryption requirements and keychain semantics
* Any referenced URLs/default servers
